# Bug Reporting

## What is a Bug?
A bug is a defect or error in the software that causes incorrect results.

## Bug Life Cycle
1. New
2. Assigned
3. Open
4. Fixed
5. Retested
6. Closed

## Severity vs Priority

- Severity → Impact of bug
- Priority → Urgency to fix

## Example
- High Severity: System crash
- High Priority: Login not working

🧩 Step-by-Step (Easy Version)
🔹 1. NEW → “Bug found”

👉 QA finds a problem and reports it

Example:
Login works even with wrong password ❌

🔹 2. ASSIGNED → “Give to developer”

👉 Manager gives this bug to a developer

🔹 3. OPEN → “Developer starts working”

👉 Developer accepts the bug and starts fixing it

🔹 4. FIXED → “Developer says it’s solved”

👉 Developer fixes code and marks bug as fixed

🔹 5. RETESTED → “QA checks again”

👉 QA tests the same bug again

🔹 6. CLOSED → “All good”

👉 If bug is really fixed → close it ✅

🔁 What if it’s NOT fixed?

👉 Then:

🔸 REOPENED → “Still broken”

QA sends it back to developer again

🔄 Full Flow (Super Simple)
NEW → ASSIGNED → OPEN → FIXED → RETESTED → CLOSED

👉 OR if issue still exists:

RETESTED → REOPENED → OPEN → FIXED → RETESTED → CLOSED
🎯 Real-Life Example
Problem:

User logs in with wrong password but still gets access

Flow:
QA finds bug → NEW
Given to dev → ASSIGNED
Dev starts → OPEN
Dev fixes → FIXED
QA checks again → RETESTED

👉 If working:
→ CLOSED

👉 If not working:
→ REOPENED

🧠 Easy Way to Remember

Think like this:

👉 Find → Give → Fix → Check → Close

🎯 One-Line Interview Answer

“Bug life cycle is the process from identifying a bug, assigning it to a developer, fixing it, retesting it, and finally closing it if resolved.”
